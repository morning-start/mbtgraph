# 📋 DocLoop — 文档质量优化循环

> **给 Agent 的第一份指令**。读完这个文件你就知道该做什么、怎么做、何时停止。
>
> 不需要问用户任何问题。所有规则、检查项、模板都在本目录下。

---

## 一、你的任务

优化 `README.md` 和 `site/src/content/docs/` 下的文档，按 **正确性 → 准确简洁 → 美观大方** 三个关卡依次提升质量。

## 二、怎么跑

### 2.1 入口

```bash
# 用户让你开始时的命令
moon run .loop  # 或直接说 "启动 DocLoop"
```

但你实际需要做的是 **阅读 `.loop/` 下的全部文件**，然后按流程执行。

### 2.2 循环流程

```
┌─────────────────────────────────────────────────┐
│  1. SCANNING                                      │
│     → 读 STATUS.md，确定本次范围                   │
│     → 读 LOOP.md（本文件）、RULES.md、TEMPLATES/* │
│     → 列出要处理的文件清单                          │
├─────────────────────────────────────────────────┤
│  2. 对每个文件运行 3 关检查                         │
│                                                     │
│  ┌── 🥇 第一关: CORRECTNESS_CHECK ──────────┐     │
│  │  检查项见 RULES.md 的 Correctness 章节       │     │
│  │  有错 → 修改 → 重新检查 → 通不过 → BLOCKED │     │
│  │  通过 → ✅ 标记 CORRECTNESS_CHECK 完成      │     │
│  └────────────────────────────────────────────┘     │
│                          ↓                          │
│  ┌── 🥈 第二关: ACCURACY_CHECK ────────────┐     │
│  │  检查项见 RULES.md 的 Accuracy 章节        │     │
│  │  有问题 → 优化 → 重新检查                   │     │
│  │  通过 → ✅ 标记 ACCURACY_CHECK 完成        │     │
│  └────────────────────────────────────────────┘     │
│                          ↓                          │
│  ┌── 🥉 第三关: AESTHETICS_CHECK ──────────┐     │
│  │  检查项见 RULES.md 的 Aesthetics 章节      │     │
│  │  有问题 → 美化 → 重新检查                   │     │
│  │  通过 → ✅ 标记 AESTHETICS_CHECK 完成      │     │
│  └────────────────────────────────────────────┘     │
│                          ↓                          │
│  3. 更新 STATUS.md                                   │
│     → 记录该文件各关卡的通过/失败状态                 │
├─────────────────────────────────────────────────┤
│  4. 重复步骤 2-3，直到所有文件全部 3 关通过          │
├─────────────────────────────────────────────────┤
│  5. 输出最终报告到 STATUS.md                         │
└─────────────────────────────────────────────────┘
```

### 2.3 关键规则

| 规则 | 说明 |
|------|------|
| **正确性不通过 → 立即暂停** | 如果发现正确性问题（代码错误、API 过期、链接 404），立即标记 `BLOCKED` 并报告用户。不要继续美化。 |
| **每次修改后必须更新 STATUS.md** | 状态文件是唯一真相来源。 |
| **不要跳过任何检查项** | 每个文件的每个关卡都要过一遍 RULES.md 的全部检查项。 |
| **保持文件现有风格** | 不要擅自重构或大规模改写。局部优化。 |

## 三、怎么知道当前进度

状态文件 `STATUS.md` 记录所有信息：

```markdown
# DocLoop 状态报告

## Meta
- 开始时间: 2026-07-07 21:00
- 当前迭代: 第 2 轮
- 整体进度: ██████░░░░ 60%
- 当前阶段: ACCURACY_CHECK

## 文件清单
| 文件 | 正确性 | 准确简洁 | 美观 | 备注 |
|------|:------:|:--------:|:----:|------|
| README.md | ✅ | ✅ | ✅ | 已全部通过 |
| site/.../bfs/index.md | ✅ | ✅ | ⏳ | 美观优化中 |
| site/.../dijkstra/index.md | ✅ | ❌ | — | 需优化表达 |
```

每次迭代结束，把这段 Markdown 更新到磁盘上。

## 四、终止条件

全部文件三个关卡都标记 `✅` → **输出报告 → 停止**。

如果遇到无法自动修复的问题（代码 API 不匹配、需要确认事实），标记 `BLOCKED` 并告诉用户原因。
