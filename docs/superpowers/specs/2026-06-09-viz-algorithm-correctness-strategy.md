# 可视化算法正确性保证策略

> **创建日期**: 2026-06-09
> **状态**: 已决策（采用 D → C 演进路线）
> **关联**: [实施计划](../plans/2026-06-09-viz-correctness-phase1.md)

---

## 1. 背景与问题

`site/` 下当前有 5 个算法可视化页面（BFS / DFS / Dijkstra / Cycle / Topo Sort），
但 [docs/ROADMAP.md](../../ROADMAP.md) 和 [astro.config.mjs](../../astro.config.mjs) 侧栏
声明的可视化目标覆盖了 ~49 个算法。

### 1.1 当前实现架构

| 层级 | 位置 | 职责 | API 形态 |
|------|------|------|----------|
| MoonBit 算法库 | `lib/algo/<category>/<algo>.mbt` | 计算最终结果 | `bfs(g, start) -> BfsResult` |
| TypeScript 步骤生成器 | `site/src/lib/algs/<algo>.ts` | 产生可视化步骤 | `generateSteps() -> Step[]` |

### 1.2 核心问题

**MoonBit 库与 TS 步骤生成器是"两份独立实现"**。这导致：

| 风险 | 严重性 | 触发条件 |
|------|:------:|----------|
| 库算法改了，TS 步骤不同步 | 高 | `lib/algo/*/bfs.mbt` 重构但 `site/src/lib/algs/bfs.ts` 未跟进 |
| TS 重写时引入 bug | 高 | 人工翻译 MoonBit→TS 时的逻辑漂移 |
| 新增算法时遗漏测试 | 中 | 加新页面时只验证"看起来对" |
| 库与演示的访问顺序不一致 | 中 | 边界条件（环、多连通分量）下两套实现走不同分支 |

### 1.3 决策范围

需要回答：

1. **是否要用 wasm 把 MoonBit 库搬到浏览器直接跑？**
2. **如何保证 TS 步骤生成器与 MoonBit 库结果一致？**
3. **如何把"正确性"工程化（自动化、可回归）？**

---

## 2. 备选方案

### 方案 A：维持现状，继续 TS 重写

| 维度 | 评价 |
|------|------|
| 实施成本 | 中（每算法 1-3 天 TS 重写） |
| 长期维护 | ❌ 极差（库改一次、TS 改一次） |
| 正确性保证 | ❌ 仅靠人工 review |
| 库 API 污染 | ✅ 无 |
| 库性能优势利用 | ❌ 无 |

**适用场景**：仅 5 个算法、长期不更新。
**结论**：不符合项目 ~49 个算法的目标，**不采用**。

### 方案 B：在 MoonBit 库中加 `xxx_with_steps` 步骤化 API

| 维度 | 评价 |
|------|------|
| 实施成本 | 高（重构 ~30+ 算法） |
| 长期维护 | ✅ 单源（库是唯一真实源） |
| 正确性保证 | ✅✅ 字节级一致 |
| 库 API 污染 | ❌ 严重（库的公共接口膨胀，且"步骤"不是生产用户关心的） |
| 库性能优势利用 | ✅ 直接跑库 |

**适用场景**：库的最终用户也需要步骤级 API。
**结论**：库的最终用户（MoonBit 开发者）只关心结果，不关心步骤。**污染库 API 得不偿失，不采用**。

### 方案 C：独立 `lib/algo_viz/` 包 + wasm 调用

| 维度 | 评价 |
|------|------|
| 实施成本 | 中-高（建包 + wasm 工具链 + 序列化层） |
| 长期维护 | ✅ 库不动，viz 包专责可视化 |
| 正确性保证 | ✅✅ 字节级一致 |
| 库 API 污染 | ✅ 无（包分离） |
| 库性能优势利用 | ✅ wasm 直接跑 MoonBit |

**关键问题**：库算法是"结果导向"，wasm 调用时需要"步骤版算法"。
本方案要求 **在 viz 包中** 写"步骤版算法"，可参考/复用库的内部算法逻辑（共享 helpers）。

**适用场景**：可视化算法数量大、性能敏感、库与 viz 长期共存。
**结论**：**作为长期目标**。但需要 wasm 工具链评估先行。

### 方案 D：wasm 仅做结果验证 + TS 生成步骤

| 维度 | 评价 |
|------|------|
| 实施成本 | 低（不需要 wasm 构建链，仅快照 + 对拍） |
| 长期维护 | ✅ 库改 → snapshot 改 → CI fail → 强制同步 |
| 正确性保证 | ✅✅ 字节级一致（最终状态） |
| 库 API 污染 | ✅ 无 |
| 库性能优势利用 | ❌ TS 跑（但站点数据规模 < 100 节点，足够） |

**关键问题**：
- 步骤生成仍由 TS 负责（保持 UI 灵活度）
- wasm 仅在每次"对拍"时跑一次，验证最终结果
- **实施不需要 wasm** —— 现阶段 moon → JSON 即可，wasm 可在阶段 2 升级

**适用场景**：先解决"正确性"问题，工具链/性能问题延后处理。
**结论**：**作为短期目标，立即执行**。

### 方案 E：完全 TS 重写 + 大量单元测试

| 维度 | 评价 |
|------|------|
| 实施成本 | 中 |
| 长期维护 | 中（TS 测试要写全） |
| 正确性保证 | ⭐⭐（TS 测试和库测试"两套"） |
| 库 API 污染 | ✅ 无 |

**适用场景**：作为 D 的退化版本（不用 moon snapshot）。
**结论**：D 已包含此方案的优点（用 moon snapshot 而非 TS 测试，更权威），D 优先。

## 3. 决策：采用 D → C 演进路线

### 3.1 决策结论

| 阶段 | 方案 | 状态 | 时间窗 |
|------|------|:----:|--------|
| **阶段 1（短期）** | **D**：快照 + 对拍测试 | 立即执行 | 2026 Q2 |
| **阶段 2（中期）** | D + 1 个算法的 wasm PoC | PoC 验证后 | 2026 Q3 |
| **阶段 3（长期）** | C：全量 wasm 化 | PoC 通过后决策 | 2026 Q4+ |

### 3.2 选择理由

**为什么先 D 后 C**：

1. **D 解决"正确性"，C 解决"性能 + 单源"**——D 是更紧迫的问题。
2. **D 的实施成本远低于 C**（不需要 wasm 工具链）。
3. **D 可以独立工作**：即使 C 永远不做，D 已经把"库改 → TS 同步"自动化了。
4. **C 之前需要 PoC**：wasm 构建、序列化、错误处理、调试工具链都是未知数，盲上 49 个算法风险大。
5. **D 给 C 铺路**：D 的"金标准快照"正好是 C 验证 PoC 的对照物。

**为什么不一步到位做 C**：

1. wasm 工具链评估需要时间（moon wasm 后端成熟度、Astro 集成、调试手段）
2. 库的 API 形态（结果导向）与可视化的需求（步骤导向）有根本差异
3. 站点数据规模 < 100 节点，TS 性能足够

### 3.3 阶段 1（D）的具体设计

#### 核心机制

```
┌─────────────────────────────────────────────────────────┐
│  MoonBit 库                                               │
│                                                            │
│  lib/algo/<category>/<algo>_test.mbt                      │
│    └─ 黑盒测试 + 快照生成（--update 模式）                  │
│                                                            │
│  输出 → lib/algo/_viz_snapshots/<algo>.json                │
│    {                                                       │
│      "graph": {"nodes": [...], "edges": [...]},            │
│      "params": {...},                                       │
│      "expected": {"order": [...], "parents": {...}, ...}    │
│    }                                                        │
└─────────────────────────────────────────────────────────┘
                          │
                          │ 复制/构建时同步
                          ▼
┌─────────────────────────────────────────────────────────┐
│  site/ 前端                                                │
│                                                            │
│  site/src/lib/algs/<algo>.test.ts                         │
│    └─ vitest: 跑 TS generateSteps() → 拿最后一步 result   │
│    └─ 与快照 JSON 做 toEqual() 对比                        │
│                                                            │
│  site/scripts/sync-snapshots.ts                           │
│    └─ 从 lib/algo/_viz_snapshots/ 复制到                  │
│       site/src/lib/algs/__snapshots__/                    │
└─────────────────────────────────────────────────────────┘
```

#### 关键约定

| # | 约定 | 说明 |
|---|------|------|
| K1 | **快照是"金标准"** | 库算法改了 → 跑 `moon test --update` 刷新 → 提交 |
| K2 | **TS 测试必须挂上** | 任何新增算法可视化页面，必须同时有 `<algo>.test.ts` |
| K3 | **对拍粒度 = 最终状态** | 访问顺序、距离、连通分量集合等结果字段 |
| K4 | **对拍不含 UI 文本** | "发现新节点！" 这类提示文案不参与对拍 |
| K5 | **同步脚本单向** | 快照从 moon 流向 site，禁止反向修改 |
| K6 | **CI 强制对拍** | 改 moon 库 → CI 跑 parity → 失败则要求同步 TS |

#### 实施边界

**D 方案不做的事**（避免范围蔓延）：

- ❌ 不做 wasm 集成
- ❌ 不重构现有 5 个算法可视化页面（除非测试 fail）
- ❌ 不改 moon 库 API
- ❌ 不为所有 49 个算法生成快照（按优先级分批）

**D 方案要做的事**：

- ✅ 建立快照生成 + 同步 + 对拍的完整管道
- ✅ 把 5 个已有算法（BFS/DFS/Dijkstra/Cycle/Topo）接入对拍
- ✅ 在 CI 加 parity job
- ✅ 在 AGENTS.md / 贡献文档加一节"正确性保证"
- ✅ 文档化"新增可视化算法的标准流程"

### 3.4 阶段 2 的触发条件

满足以下任一条件，进入 PoC：

1. 站点数据规模超过 100 节点，TS 性能出现明显问题
2. 库的算法 API 经常改，每次改都要触发 CI parity 失败太多次
3. 团队对 wasm 工具链的掌握成熟

**PoC 范围**：仅 1 个算法（建议 BFS），跑通 moon wasm → Astro 加载 → 调用 → 渲染。

**PoC 通过标准**：

- ✅ wasm 在浏览器加载成功（< 500ms 初始化）
- ✅ TS 调用 wasm 拿结果，TS 拿原 moon 库结果做对拍
- ✅ 两者字节级一致
- ✅ wasm 包大小可接受（< 1MB gzip）
- ✅ Astro 集成顺畅（构建时拷贝 wasm 到 public/）

**PoC 不通过时的兜底**：

- 维持 D 不变，仅修复发现的工具链问题
- 评估是否有"渐进式 wasm 化"（如先做无依赖算法、避开有 JS 互操作复杂的）

## 4. 风险登记

| # | 风险 | 影响 | 缓解 |
|---|------|------|------|
| R1 | 快照生成测试与"真实生产用例"不一致 | 中 | 复用 `lib/algo/*/traversal_test.mbt` 已有的固定小图 |
| R2 | TS 步骤生成器与 moon 库走的不是同一算法路径 | 中 | 快照覆盖"典型用例 + 边界用例 + 退化用例"三类 |
| R3 | 同步脚本遗漏，新算法忘了挂测试 | 中 | CI 失败 + 文档化流程 + review checklist |
| R4 | moon 测试 → 快照的 --update 模式被滥用 | 低 | PR review 时强制要求解释 snapshot diff |
| R5 | vitest 在 site 项目里未启用，需补基础设施 | 低 | 任务 1 一次性装好 |
| R6 | 序列化格式不统一（moon String vs TS JSON） | 中 | 任务 2 制定明确的 `expected` schema 规范 |

## 5. 成功度量

| 指标 | 当前 | D 完成后目标 | 衡量方式 |
|------|:----:|:------------:|----------|
| 可视化算法数量 | 5 | 5（仅接入对拍） | `ls site/src/pages/visualizations/*.astro` |
| 算法演示与库结果一致率 | 人工保证 | **100% 自动化对拍** | CI parity job 通过率 |
| 库改 → TS 同步延迟 | 数天-数周 | **PR 内必现** | CI fail → 修复 → merge |
| 新增算法可视化的标准流程 | 模糊 | **文档化的 5 步流程** | CONTRIBUTING.md 章节 |
| wasm 调用 | 0 | 0（阶段 1 不做） | - |
